# SelfClap プロジェクト憲法

## 基本原則

### 1. シンプルさ第一
- コードベースを最小限かつ保守可能に保つ
- 複雑なアーキテクチャよりもシンプルな解決策を優先
- すべての機能は明確なユーザー価値を持つこと

### 2. ローカルファースト哲学
- すべてのデータを `~/.selfclap/` にローカル保存
- 外部サービス不要（Claude Code統合を除く）
- ユーザーが自分のデータを完全に所有
- プライバシー重視の設計 - データは端末外に出ない

### 3. Claude Code統合前提の設計
- **すべての操作はClaude Code CLI上から実行されることを前提とする**
- ユーザーはClaude Codeと会話しながらSelfClapを使用
- Claude CodeがAI機能（褒め、励まし、分析）を担当
- SelfClapはデータ管理・可視化に特化したツール
- APIキー不要 - Claude Code Maxプラン利用者がそのまま使える
- Claude Codeとの自然な会話フローを重視した設計

### 4. 開発者体験
- 明確で親しみやすいエラーメッセージ
- 直感的なコマンド名（`clap diary write`、`clap task add`）
- 段階的な開示 - 基本機能はすぐに動作
- 高度な機能は `--help` で発見可能
- Claude Codeが解釈しやすい構造化出力

### 5. コード品質基準
- すべての関数シグネチャに型ヒント
- パブリックAPIにはdocstring
- コア機能のユニットテスト
- 関数は小さく焦点を絞る（50行未満）

### 6. データ整合性
- 適切なスキーマ制約を持つSQLite
- データ変更はアトミック操作
- エラー時の適切な処理
- 常にユーザーデータを保護（確認なしの破壊的操作は行わない）

### 7. 可視化とフィードバック
- 色と書式設定を使ったRichなターミナル出力
- 統計とレポートの明確な視覚階層
- 達成時の祝福フィードバック（連続記録、マイルストーン）
- あらゆるターミナルで動作するASCIIベースの可視化

### 8. 拡張性
- モジュラーアーキテクチャ（cli、database、visualization分離）
- ユーザー設定はYAML経由で管理
- 既存機能を壊さずに新コマンドを追加しやすい設計

### 9. ユーザーのエンパワーメント
- ポジティブな強化に焦点
- 大きな成果だけでなく小さな勝利も祝う
- 進捗を可視化し実感できるようにする
- 日常的な使用と習慣形成のための設計

### 10. パフォーマンス
- 高速な起動時間（シンプルなコマンドで100ms未満）
- 効率的なクエリ（インデックス付きデータベースフィールド）
- 最小限の依存関係
- オフラインで動作

## UI/UX原則

### 日本語ファースト
- すべてのメッセージ、出力、ドキュメントは日本語
- 新卒エンジニアに親しみやすい表現
- 専門用語は適度に、過度に難しくしない

### 励ましと承認
- 批判的なトーンは避ける
- 小さな努力も認める
- 継続を称賛する
- 前向きな言葉選び

### 透明性
- データがどこに保存されるか明示
- コマンドの実行結果を分かりやすく表示
- エラーが起きたら次に何をすべきか提示
